# Vendoring JS bundles in non-Node Django projects

## When to use

Adding a JavaScript library (editor, syntax highlighter, charting lib, drag-handle helper, etc.) to a Django project that does **not** have a Node toolchain in CI/Docker/day-to-day. The library is published on npm and its docs say "use a bundler" — but you don't want to add esbuild/vite/webpack to the Docker image just for one widget.

**Decision rule:** vendor the bundle.

## Why vendor instead of adding Node

- Django + HTMX + Alpine + Bulma is a server-rendered stack. The runtime image is Python + nginx; introducing Node solely to bundle one library doubles the image complexity, the CI surface, and the routine-development friction.
- npm packages bring transitive-dependency churn (Dependabot noise, `npm audit` alerts you don't want, `package-lock.json` merge conflicts).
- The vendored file is a **single artefact** in your version control — diffable, reviewable, rollback is `git revert`.
- Upgrade ritual is bounded ("rebuild when bumping the library version"), unlike a Node toolchain which constantly demands attention.

**When NOT to vendor:**

- The library mandates a build step that uses many of its source modules (very rare; most modern libraries publish a usable bundle).
- The library shares dependencies with other libraries you're already bundling (transitive de-dup matters → you genuinely need a bundler).
- Your project is React-/Vue-first and you already have Node — vendoring is then over-engineering.

## The pattern

Layout:

```
web/<app>/static/vendor/<library>/
  ├── <library>.min.css       # ← committed pre-built file
  └── <library>.min.js        # ← committed pre-built file

web/<app>/static/<app>/js/<library>-widget.js  # ← integration glue (handcrafted)
```

Templates load via Django's `{% static %}`:

```django
<link rel="stylesheet" href="{% static 'vendor/easymde/easymde.min.css' %}">
<script src="{% static 'vendor/easymde/easymde.min.js' %}"></script>
<script src="{% static '<app>/js/easymde-widget.js' %}" defer></script>
```

> **App-shell caveat.** These eager `<script>` tags live in `{% block extra_js %}`, *outside* an
> HTMX shell-swap region — so on cross-surface shell-nav they never re-execute, and a heavy
> per-surface bundle silently fails to load (renders nothing until a hard refresh). When that
> bites, load the bundle on-nav instead: see
> [`LAZY_LOAD_HEAVY_ASSETS_ON_HTMX_NAV.md`](LAZY_LOAD_HEAVY_ASSETS_ON_HTMX_NAV.md).

Forms hook into the integration glue via data attributes:

```python
# forms.py
class ArticleForm(forms.ModelForm):
    class Meta:
        widgets = {
            'contents': forms.Textarea(attrs={'rows': 12, 'data-easymde': ''}),
        }
```

The integration glue (`easymde-widget.js`) auto-discovers `[data-easymde]` textareas on `DOMContentLoaded` + after HTMX swaps, mounts the editor, handles teardown on `htmx:beforeSwap`, wires uploads to `/api/<app>/attachments/`.

## Producing the vendor file

### Path A: Library publishes a CDN/UMD distribution

Easiest case. Download once from unpkg.com / jsDelivr / the library's own dist:

```bash
mkdir -p web/<app>/static/vendor/<library>/
curl -L -o web/<app>/static/vendor/<library>/<library>.min.js \
  https://unpkg.com/<library>@<pinned-version>/dist/<library>.min.js
# repeat for .css if applicable
git add web/<app>/static/vendor/<library>/
```

Document the `unpkg.com` URL in a header comment of the integration glue file so the next maintainer knows where to refresh from.

### Path B: Library is ESM-only, requires bundling (e.g., TipTap, ProseMirror)

Spin up a disposable Node container for a one-time build. No Node on the host or in CI/Docker:

```
tools/build-<library>-bundle/
  ├── package.json              # pinned library + extension versions
  ├── package-lock.json         # generated by npm ci
  ├── build.js                  # ~30-line esbuild script
  ├── src/
  │   └── index.js              # entry: imports library, exports window.<Global>
  ├── test/                     # optional — Node-side fixture tests
  │   └── roundtrip.js
  └── README.md                 # when to rebuild + how

tools/build-<library>-bundle.sh # wrapper script that does the docker run
```

The wrapper script:

```bash
#!/usr/bin/env bash
# tools/build-<library>-bundle.sh
set -euo pipefail

cd "$(dirname "$0")/build-<library>-bundle"
docker run --rm \
  -v "$PWD:/work" \
  -w /work \
  node:22-alpine \
  sh -c 'npm ci --silent && node build.js'

cp dist/<library>-bundle.js \
   ../../web/<app>/static/vendor/<library>/<library>-bundle.js

# Architect F9 pattern: write BUILD_INFO.txt for drift detection
LOCK_HASH=$(sha256sum package-lock.json | cut -d' ' -f1)
BUNDLE_HASH=$(sha256sum ../../web/<app>/static/vendor/<library>/<library>-bundle.js | cut -d' ' -f1)
SIZE=$(wc -c < ../../web/<app>/static/vendor/<library>/<library>-bundle.js)
TS=$(date -u +%Y-%m-%dT%H:%M:%SZ)
cat > ../../web/<app>/static/vendor/<library>/BUILD_INFO.txt <<EOF
package_lock_sha256=${LOCK_HASH}
bundle_sha256=${BUNDLE_HASH}
built_at=${TS}
bundle_bytes=${SIZE}
EOF
```

Run it:

```bash
bash tools/build-<library>-bundle.sh
git add web/<app>/static/vendor/<library>/ tools/build-<library>-bundle/
git commit -m "feat(<app>): vendor <library>@<version>"
```

The disposable container leaves zero host artefacts. Day-to-day development has no Node.

### Path C: Browser ESM via importmap

Possible (`<script type="module">` from esm.sh / jspm.io), but two strikes against it for Django+HTMX projects:

1. Runtime CDN dependency is production-fragile (CDN goes down → editor broken).
2. Multiple round-trips on first page load before the editor is interactive.

Skip unless you have a strong reason; vendoring (A or B) is the better default.

## Drift-protection guardrail (recommended for Path B)

The `BUILD_INFO.txt` hash trick:

```python
# tests/test_vendor_bundle_integrity.py
import hashlib
from pathlib import Path

def test_bundle_matches_lockfile_and_build_info():
    lock_path = Path("tools/build-<library>-bundle/package-lock.json")
    bundle_path = Path("web/<app>/static/vendor/<library>/<library>-bundle.js")
    info_path = Path("web/<app>/static/vendor/<library>/BUILD_INFO.txt")

    expected_lock = hashlib.sha256(lock_path.read_bytes()).hexdigest()
    expected_bundle = hashlib.sha256(bundle_path.read_bytes()).hexdigest()
    info = dict(line.strip().split("=", 1) for line in info_path.read_text().splitlines() if "=" in line)

    assert info["package_lock_sha256"] == expected_lock, (
        "Lockfile out of sync with BUILD_INFO. "
        "Run `bash tools/build-<library>-bundle.sh` and commit the output."
    )
    assert info["bundle_sha256"] == expected_bundle, (
        "Bundle file has been modified outside the build script (hand-edit, "
        "merge resolution, IDE auto-format, or partial rebuild). "
        "Run `bash tools/build-<library>-bundle.sh` and commit the output."
    )
```

Catches: someone hand-edits the bundle (bundle_sha256 mismatch), rebuilds without committing the new lockfile (lockfile mismatch), or upgrades the library and forgets to rebuild (lockfile mismatch). Does not catch a coordinated forgery that rewrites both the bundle and BUILD_INFO.txt to match — that's a code-review concern, not a unit-test concern.

## Integration-glue contract

Independent of the bundle, the integration glue (`<library>-widget.js`) is your project's integration
boundary — keep it OUT of the bundle (the bundle stays pure-library).

The glue follows the shared **[HTMX widget lifecycle](HTMX_WIDGET_LIFECYCLE.md)** contract: discover
`[data-<library>]` elements + (re-)mount on `htmx:afterSwap`, idempotent mount tracked in a
`Map<HTMLElement, Handle>`, teardown (`handle.destroy()`) on `htmx:beforeSwap`/`beforeSettle` to avoid
leaking detached editor cores. On top of that, two concerns the lifecycle contract doesn't cover:

- **Race-safe uploads (if applicable)** — `AbortController` per upload; abort on destroy; check `if (!activeEditors.has(el)) return` in the success path before applying results to the editor (defends against responses arriving after teardown).
- **Form integration** — sync widget state to the underlying `<input>` / `<textarea>` so Django form submit picks up the value.
